
# 《接口设计与交易所适配》模板

## 1. 文档信息

### 1.1 文档名称

接口设计与交易所适配说明书

### 1.2 适用版本

V1.0

### 1.3 适用项目

AI增强型虚拟货币网格交易系统

### 1.4 适用范围

* 首发交易对：DOGEUSDT
* 首发场景：合约网格 / 可扩展到现货
* 模块范围：行情接入、账户接入、订单执行、状态同步、风控联动、交易所适配层

### 1.5 文档目标

本文档用于定义：

* 系统内部接口边界
* 与交易所交互的标准能力
* 交易所适配层抽象设计
* 数据模型映射规则
* 错误处理与重试规则
* 幂等、签名、限流、时序一致性要求
* 多交易所扩展规范

---

## 2. 设计目标

### 2.1 总体目标

为策略引擎、AI模块、风控模块提供统一、稳定、可扩展的交易接口，使业务逻辑不依赖某一家交易所的具体 API 细节。

### 2.2 具体目标

* 屏蔽不同交易所 API 差异
* 对行情、下单、撤单、账户、持仓进行统一抽象
* 保障下单链路可追踪、可重试、可恢复
* 支持 WebSocket + REST 混合模式
* 为未来多交易所扩展预留适配层

### 2.3 设计原则

* 先抽象、后适配
* 内部统一协议优先
* 幂等优先
* 查询校正优于本地猜测
* 风控优先于执行
* 失败可恢复、结果可追踪

---

## 3. 接口总体架构

## 3.1 分层设计

建议拆为以下层级：

* 策略层
* 风控层
* 交易网关层
* 交易所适配层
* REST / WebSocket 客户端层
* 状态同步层

## 3.2 分层职责

### 3.2.1 策略层

* 生成目标订单意图
* 接收行情和账户状态
* 不直接依赖交易所原始接口

### 3.2.2 风控层

* 下单前校验
* 风险动作拦截
* 执行限制输出

### 3.2.3 交易网关层

* 接收统一下单请求
* 调用适配层
* 做请求跟踪、幂等控制、错误封装

### 3.2.4 交易所适配层

* 将统一模型转换为交易所具体协议
* 将交易所返回结果映射回系统内部模型

### 3.2.5 客户端层

* REST请求
* WebSocket连接
* 签名
* 限流
* 重连

### 3.2.6 状态同步层

* 定期同步账户、持仓、订单
* 修正本地状态偏差
* 做断线恢复

## 10. 订单状态映射设计

## 10.1 内部统一状态建议

* `NEW`
* `PARTIALLY_FILLED`
* `FILLED`
* `CANCELED`
* `REJECTED`
* `EXPIRED`
* `PENDING`
* `UNKNOWN`

## 10.2 映射原则

不同交易所订单状态命名不同，适配层应统一映射到内部标准状态。

## 10.3 特殊要求

* 若 WebSocket 先到、REST 后到，以最新可信状态更新
* 若状态冲突，优先走查询确认
* `UNKNOWN` 状态不得直接驱动补单，必须先查询校验

---

## 11. 下单链路设计

## 11.1 标准流程

1. 策略生成订单请求
2. 风控校验
3. 精度与交易规则校验
4. 生成幂等标识
5. 发送交易所请求
6. 记录请求日志
7. 接收响应并更新订单状态
8. 若状态不明确，发起查询确认
9. 进入订单跟踪

## 11.2 幂等要求

每一笔订单必须具备唯一 `client_order_id`，建议包含：

* 策略ID
* 网格层级
* 时间戳
* 重试序号

## 11.3 重试原则

* 网络异常可重试
* 参数错误不可重试
* 不确定是否下单成功时，不得直接重复发单，必须先查询

---

## 12. 撤单链路设计

## 12.1 标准流程

1. 发起撤单请求
2. 记录撤单意图
3. 等待交易所响应
4. 若超时，则主动查询订单状态
5. 更新订单状态
6. 若撤单失败但订单已成交，进入成交处理流程

## 12.2 风险点

* 撤单时订单已成交
* 撤单响应丢失
* 网络抖动导致重复撤单

## 12.3 处理原则

* 撤单失败不代表订单仍存在
* 以最终查询结果为准
* 撤单结果必须进入审计日志

---

## 13. 状态同步设计

## 13.1 同步目标

确保本地状态与交易所真实状态一致。

## 13.2 同步对象

* 账户余额
* 持仓
* 未完成订单
* 最近成交
* 杠杆状态

## 13.3 同步机制

建议组合使用：

* WebSocket 实时推送
* REST 定时校正
* 异常时强制全量同步

## 13.4 恢复场景

以下场景必须触发强制同步：

* 服务启动
* WebSocket 重连
* 下单返回不确定
* 风控进入高危状态
* 模式切换
* 策略恢复运行

---

## 14. 签名与鉴权设计

## 14.1 目标

保障请求合法、安全、可审计。

## 14.2 要求

* API Key 与 Secret 通过安全配置中心管理
* 不得硬编码在代码中
* 签名过程由适配层封装
* 日志中不得泄露敏感凭证
* 支持测试与生产环境隔离

## 14.3 安全建议

* 分环境密钥隔离
* 最小权限原则
* 定期轮换
* 对关键操作启用额外审计

---

## 15. 限流与重试设计

## 15.1 限流目标

防止触发交易所 API 限流导致整体不可用。

## 15.2 设计建议

* REST 请求分组限流
* 不同接口不同配额
* 查询和下单配额分离
* 高优先级接口预留配额

## 15.3 重试分类

### 可重试

* 网络超时
* 临时服务不可用
* 短时连接断开

### 不可重试

* 参数非法
* 签名错误
* 权限不足
* 精度错误
* 风控拒绝

## 15.4 重试策略

* 指数退避
* 限制最大重试次数
* 对不确定请求先查询再决定是否重试

---

## 16. 错误码与异常封装设计

## 16.1 目标

将交易所原始错误统一为内部标准异常，便于风控、策略和监控处理。

## 16.2 内部错误分类建议

* `VALIDATION_ERROR`
* `AUTH_ERROR`
* `RATE_LIMIT_ERROR`
* `NETWORK_ERROR`
* `EXCHANGE_TEMP_ERROR`
* `ORDER_REJECTED`
* `ORDER_UNKNOWN`
* `STATE_SYNC_ERROR`
* `UNSUPPORTED_OPERATION`

## 16.3 错误对象建议字段

* `error_code`
* `error_type`
* `message`
* `retryable`
* `raw_code`
* `raw_message`
* `request_id`
* `timestamp`

---

## 17. 精度与规则适配设计

## 17.1 目标

确保所有价格和数量满足交易所约束。

## 17.2 规则类型

* 价格最小变动单位
* 数量最小步长
* 最小下单量
* 最小名义价值
* 最大杠杆
* 支持订单类型

## 17.3 处理原则

* 下单前统一格式化
* 格式化后再次校验
* 不允许静默截断导致策略偏差不可见
* 若格式化后偏差超过可接受阈值，应拒单并告警

---

## 18. 时间与时序设计

## 18.1 目标

降低因延迟、重放、时钟差异带来的状态混乱。

## 18.2 要求

* 所有事件统一使用系统标准时间格式
* 保存本地接收时间和交易所事件时间
* 对关键操作生成请求ID和链路ID
* 处理乱序消息时以状态机规则和查询校验为准

## 18.3 时序字段建议

* `event_time`
* `exchange_time`
* `received_time`
* `processed_time`

## 21. 多交易所扩展规范

新增交易所时不得修改策略逻辑，只新增或扩展 Adapter。

## 21.2 接入步骤

1. 实现适配器类
2. 完成字段映射
3. 完成错误码映射

def submit_order(order_request):
    validate_order_request(order_request)
    risk_check(order_request)
    normalize_by_symbol_rules(order_request)

    existing = idempotency_store.find(order_request.client_order_id)
    if existing:

    try:
